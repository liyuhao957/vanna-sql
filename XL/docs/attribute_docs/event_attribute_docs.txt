事件和属性绑定关系说明：

# 全局属性（所有事件共有）
以下属性是所有事件类型(imp、clk、cls、send2等)共有的：
- launchId: 吊起id (string)
- uniqueId: 设备标识，用于AB测试流量染色 (string)
- userId: 用户id (string)
- sourceCid: 源头cid (string)
- channelId: 渠道id (string)
- packageName: 包名 (string)
- brand: 厂商品牌 (string)
- PCCSDKVersion: PCC_SDK版本 (string)
- traceId: 媒体事件跟踪标识 (string)
- accountId: 账户 (string)
- ggBodyLevel1: 广告层级1 (string)
- ggBodyLevel2: 广告层级2 (string)
- clientTime: 设备当前时间戳 (bigint)
- onceStayTime: 当次累计使用时长 (int)
- event: 事件类型，如imp、clk、cls、send2等 (string)
- timedt: 事件发生的时间 (timestamp)

# 1. 策略配置事件(config)
config事件表示策略配置上报，在策略命中、下发广告配置时上报，由服务端处理，有以下特有属性：
- clientParams: 客户端透传信息 (object)
- securityId: 安全策略id (string)
- securityName: 安全策略名称 (string)
- securityParams: 安全策略参数详情 (object)
- sceneV4Id: V4场景策略id (string)
- sceneV4Name: V4场景策略名称 (string)
- sceneV4AB: V4场景策略的AB分组 (string)
- sceneV4Params: V4场景策略参数详情 (object)
- userGroupId: 当前用户集合id (string)
- userGroupName: 当前用户集合名称 (string)
- userGroupParams: 用户分层详情 (object)
- mediaId: 我方媒体（应用）id (string)
- UA: 用户ua (string)
- IP: 用户ip (string)
- cityName: 城市名 (string)
- regionName: 省名称 (string)

# 事件间关系
- imp、clk、cls和send2事件通过reqId关联，表示同一次广告请求的不同阶段
- 一个reqId可能对应一次imp，多次clk，和一次cls
- 当分析点击率(CTR)时，应统计imp和clk事件的比率
- 分析广告效果时，常用isAct(是否激活)和isD1Act(是否次日留存激活)指标
- appLaunch、pageview、pageLeave、appView和appLeave事件可用于分析用户在应用内的行为路径 
- setPullBackTimer、realPullBack、offPullBackTimer和successPullBack事件用于分析应用提醒功能的效果
- clickComplain、submitComplain、complain1、complain2和complain3事件可以用于分析用户投诉路径和原因

## 用户完整行为路径
- 用户会话路径: appLaunch → appView → pageview → {各类交互事件} → pageLeave → appLeave，通过launchId关联同一会话
- 用户停留时间: 可通过计算appLaunch和appLeave之间的时间差获得，或计算pageview和pageLeave之间的时间差获得页面停留时间
- 页面切换频率: 通过分析同一launchId下pageview和pageLeave事件的频次可衡量用户活跃度
- 会话质量评估: 单次会话中的广告互动次数(imp、clk)与页面查看数量的比率可评估会话质量

## 广告展示关联分析
- 广告请求漏斗: req2 → send2 → imp → clk → cls，各阶段通过reqId关联，可计算转化率
- 广告质量评估: 通过比较不同reqId对应的cls.isEnded=true的比率，评估广告内容质量
- 广告类型效果: 分析不同ggIdType的req2→clk转化率，评估不同广告类型效果
- 广告投放优化: 通过分析sceneV4Id与imp、clk事件的关联，优化不同场景的广告投放策略
- 广告收益评估: 通过分析winPrice与广告互动(imp、clk)的关系，评估收益与用户体验平衡

## 投诉流程分析
- 投诉完整路径: clickComplain → complain1 → complain2 → complain3 → submitComplain → {complainQuit/complainContinue}
- 投诉原因分类: 通过complainReason字段分析投诉热点，与reqId关联可追溯到具体广告
- 投诉转化率: 计算clickComplain到submitComplain的转化率，分析投诉流失点
- 投诉后行为: 分析complainQuit和complainContinue的比例，评估投诉对用户留存的影响
- 投诉源分析: 将clickComplain与之前的imp事件关联，识别引发投诉的广告特征
- 客服效果分析: 通过分析clickOnlineService和clickHotline后的用户行为，评估客服解决效果

## 设置与提醒功能关联
- 设置访问路径: clickSetEntry → goSet，分析用户对设置的访问频率与操作系统版本的关系
- 提醒效果评估: setPullBackTimer → {offPullBackTimer/realPullBack} → {successPullBack/失败}，分析不同way的拉回成功率
- 提醒时机优化: 分析setPullBackTimer与用户appLeave时间的关系，优化提醒触发时机
- 系统设置影响: 分析goSet后用户行为变化，评估系统设置对应用使用的影响

## 错误与异常分析
- 错误溯源: appError事件与之前的用户操作序列关联，识别可能导致错误的操作路径
- 广告错误分析: 将err事件与req2、send2关联，分析广告加载失败的原因和影响
- 错误对留存影响: 分析appError后用户的appLaunch频率变化，评估错误对留存的影响
- 设备兼容性: 通过brand、model与appError的关联，分析设备兼容性问题

## 跨事件分析方法
- 会话归因: 使用launchId将同一会话内的所有事件关联起来进行整体分析
- 用户画像构建: 通过userid关联用户的所有事件，构建长期行为画像
- 广告效果评估: 使用reqId将广告相关事件(req2、send2、imp、clk、cls)关联为完整链路
- 投诉原因归因: 将submitComplain的投诉与之前的广告展示(imp)通过时间窗口和用户id关联
- A/B测试分析: 通过uniqueId分组分析不同功能、界面对用户行为的影响
